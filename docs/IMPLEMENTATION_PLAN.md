# Implementation Plan — AI Development Studio v1 (Issue #6)

Статус: план, реализация не начата. Ни один этап ниже не выполнен.

## Контекст и известные ограничения

- Ранее в этом репозитории уже велась работа над частью scope Issue #6:
  - PR #3 (`feature/issue-2-template-repository`, Issue #2 «Template Repository v1»)
    добавлял Issue-шаблоны, PR-шаблон, ADR-шаблон, `templates/AGENTS.md`,
    `templates/README.md`, пример структуры проекта и CI-workflow
    `markdown-lint.yml`. PR закрыт без merge — эти файлы существуют только в
    ветке `feature/issue-2-template-repository`, в `main` их нет.
  - PR #5 (`hotfix/markdownlint-baseline`, Issue #4) чинил нарушения
    markdownlint в `AI_OS.md` и `.github/PULL_REQUEST_TEMPLATE.md`. Тоже закрыт
    без merge.
  - `main` сейчас не содержит ни CI-workflow, ни GitHub-шаблонов, ни
    markdownlint-чистого `AI_OS.md`/`README.md`/`CHANGELOG.md`.
- Следствие: этап 1 ниже — не новая работа, а обязательное условие, без
  которого CI для всех последующих этапов будет либо не запускаться (нет
  workflow), либо падать на предсуществующем формате файлов, не относящихся к
  текущему PR.

## Обязательное ограничение

`AI_OS.md` не изменяется по смыслу ни на одном этапе. Разрешены только
чисто форматные правки (перенос строк, пустые строки), если понадобятся для
прохождения markdownlint — без изменения текста правил.

## Этапы реализации

### Этап 1 — Markdownlint-baseline + CI-workflow в `main`
Перенести в `main`:
- `.github/workflows/markdown-lint.yml` (из `feature/issue-2-template-repository`,
  без изменений — pinned SHA действий уже проверены по официальным репозиториям
  в предыдущей работе);
- форматные исправления `AI_OS.md`, `.github/PULL_REQUEST_TEMPLATE.md`,
  `README.md`, `CHANGELOG.md` под правила markdownlint (только формат, не
  смысл).

**Зависимости:** нет — это базовый этап, всё остальное зависит от него.
**Отдельный PR:** да, отдельный (аналог PR #5 + добавление workflow).
**Параллельно с чем-либо:** нет, должен закрыться первым.
**Проверки после этапа:**
- GitHub Actions `lint` зелёный на этом PR;
- diff не содержит изменений смысла `AI_OS.md` (построчное сравнение с текущей
  версией в `main`, разрешены только переносы/пустые строки);
- независимый ревьюер подтверждает, что скопированный workflow идентичен
  версии, ранее сверенной с официальными репозиториями `actions/checkout` и
  `DavidAnson/markdownlint-cli2-action`.

### Этап 2 — GitHub-шаблоны и архитектурные шаблоны
Восстановить в `main` (перенести из закрытого PR #3, при необходимости
адаптировать под текущее состояние `main`):
- `.github/ISSUE_TEMPLATE/bug_report.md`, `feature_request.md`;
- `templates/adr/ADR_TEMPLATE.md`;
- `templates/EXAMPLE_PROJECT_STRUCTURE.md`.

**Зависимости:** Этап 1 (нужен зелёный CI-baseline, иначе новый PR будет падать
на чужих ошибках).
**Отдельный PR:** да.
**Параллельно с чем-либо:** можно параллельно с Этапом 3 (структура проекта),
если ADR- и Issue-шаблоны не пересекаются файлово со структурой — на практике
пересечений нет, конфликтов при рестайне не ожидается.
**Проверки после этапа:**
- CI зелёный;
- `AI_OS.md` не затронут (сравнение SHA);
- каждый шаблон открыт и визуально проверен в GitHub UI (Issue-форма
  создаётся корректно, ADR и PR шаблоны рендерятся без разметочных ошибок).

### Этап 3 — Стандартная структура проекта
Определить и задокументировать (например, в `templates/`):
- рекомендуемое дерево каталогов нового проекта (расширение
  `EXAMPLE_PROJECT_STRUCTURE.md` из Этапа 2);
- `templates/README.md`, `templates/AGENTS.md` — актуализировать под финальную
  структуру.

**Зависимости:** Этап 1 обязательно; Этап 2 логически (структура ссылается на
шаблоны из Этапа 2), но при аккуратной формулировке ссылок можно вести
параллельно с Этапом 2 в отдельных ветках с последующей быстрой синхронизацией
перед merge обеих PR.
**Отдельный PR:** да.
**Параллельно с чем-либо:** с Этапом 2 (см. выше) и с Этапом 6 (независимый
ревью — процессный документ, не пересекается файлово).
**Проверки после этапа:**
- CI зелёный;
- все пути, упомянутые в структуре, реально существуют в репозитории после
  Этапа 2 (нет ссылок на несуществующие шаблоны);
- независимый ревьюер проверяет соответствие структуре, заявленной в Issue #6.

### Этап 4 — Project Cockpit
Создать центральный документ/дашборд проекта (например,
`docs/PROJECT_COCKPIT.md`), агрегирующий:
- ссылки на `AI_OS.md`, шаблоны, инструкции для агентов (Этап 5);
- текущий статус инициализации нового проекта;
- чек-лист запуска.

**Зависимости:** Этап 3 (нужна финальная структура, чтобы Cockpit ссылался на
реальные пути), Этап 1.
**Отдельный PR:** да.
**Параллельно с чем-либо:** нет — Cockpit по определению агрегирует результат
предыдущих этапов, имеет смысл делать после того, как структура (Этап 3)
стабилизировалась.
**Проверки после этапа:**
- CI зелёный;
- все ссылки в Cockpit ведут на существующие файлы (проверка вручную или
  скриптом-линком, если появится в проекте);
- независимый ревьюер подтверждает, что Cockpit не содержит инструкций,
  противоречащих `AI_OS.md`.

### Этап 5 — Инструкции для GPT, Claude Code, Codex
Три отдельных документа (например, `docs/agents/GPT.md`,
`docs/agents/CLAUDE_CODE.md`, `docs/agents/CODEX.md`), каждый — адаптация
`AI_OS.md` и шаблонов под конкретный инструмент, без дублирования сути
принципов (только ссылки + специфика инструмента).

**Зависимости:** Этап 1 (стабильный `AI_OS.md`). Желательно после Этапа 2/3
(чтобы ссылаться на финальные шаблоны), но содержательно не блокируется ими.
**Отдельный PR:** да, рекомендуется три отдельных PR (по одному на GPT, Claude
Code, Codex) — так независимый ревью каждого инструмента идёт изолированно и
правки в одном не блокируют мерж остальных.
**Параллельно с чем-либо:** да — все три документа независимы друг от друга и
могут писаться параллельно; также параллельно с Этапом 6.
**Проверки после этапа (на каждый из трёх PR отдельно):**
- CI зелёный;
- документ не противоречит `AI_OS.md` (независимый ревьюер сверяет построчно);
- нет расхождений между тремя документами в описании общих принципов (сверка
  делается после того, как все три готовы, отдельным шагом ревью).

### Этап 6 — Процесс независимого ревью
Задокументировать процесс independent review (например,
`standards/INDEPENDENT_REVIEW.md`): кто может ревьюить, что проверяется, как
фиксируется результат.

**Зависимости:** Этап 1 только (ссылается на принцип Independent Review из
`AI_OS.md`, который уже существует и не меняется).
**Отдельный PR:** да.
**Параллельно с чем-либо:** да — не пересекается файлово ни с одним другим
этапом, может идти параллельно с Этапами 2, 3, 5.
**Проверки после этапа:**
- CI зелёный;
- документ проверен независимым ревьюером (в идеале — тем, кто не писал
  документ, для проверки применимости процесса на практике);
- соответствие Acceptance Criteria Issue #6 в части «проведено независимое
  ревью».

### Этап 7 — Стартовая документация нового проекта
Финальный интеграционный документ (например,
`docs/START_HERE.md` в контексте шаблона) — точка входа для человека или
AI-агента, создающего новый проект из этого Template Repository. Ссылается на
Cockpit (Этап 4), инструкции агентов (Этап 5), структуру (Этап 3).

**Зависимости:** Этапы 2, 3, 4, 5, 6 — это интеграционный этап, требующий,
чтобы всё перечисленное уже существовало в `main`.
**Отдельный PR:** да, последний перед итоговой валидацией.
**Параллельно с чем-либо:** нет — по определению идёт после всех остальных.
**Проверки после этапа:**
- CI зелёный;
- ручной таймированный прогон «создание нового проекта с нуля» — фиксируется
  фактическое время, сверяется с критерием «менее 5 минут»;
- подтверждение, что все инструкции в документе берутся из репозитория
  (никаких ссылок на контекст чата или внешние источники, не входящие в
  репозиторий).

### Этап 8 — Итоговая валидация против Acceptance Criteria
Не PR с новым кодом, а контрольная точка: сверка каждого пункта Acceptance
Criteria Issue #6 (Template Repository пригоден к использованию, < 5 минут на
новый проект, все инструкции из репозитория, чат не источник истины, Draft PR
открыт, независимое ревью проведено, ничего не смёржено).

**Зависимости:** Этапы 1–7 полностью завершены.
**Отдельный PR:** нет — это финальный чек-лист/комментарий к Issue #6, либо
финальный Draft PR, суммирующий все предыдущие (без merge).
**Параллельно с чем-либо:** нет.
**Проверки:** построчная сверка с разделом Acceptance Criteria Issue #6.
